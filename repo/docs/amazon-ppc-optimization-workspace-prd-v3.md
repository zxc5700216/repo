# Amazon PPC Optimization Workspace
## Product Requirements Document
### Version 3.0

---

## 1. Document Positioning

This document is the master PRD for the Amazon PPC Optimization Workspace.

It replaces the earlier document model centered on:

- Workbook
- Sheet
- Table
- Row

The new product model is centered on:

- Campaign
- Workspace
- Keyword / Product Target
- Rule
- Draft
- Execution

In the new architecture, `Workbook` is no longer treated as the product's primary interaction model. It is only an input carrier. The actual product experience begins after the system converts raw workbook data into campaign workspaces that can be analyzed, optimized, simulated, reviewed, and exported.

This PRD is written for:

- Product design
- Frontend architecture
- Rule engine design
- Data model design
- Local-first SaaS implementation
- Future AI optimization extension
- Code generation guidance for Codex and other engineering copilots

## 2. Product Vision

### 2.1 Product Position

Amazon PPC Optimization Workspace is a workspace-based advertising optimization system for Amazon sellers and operators.

It is not an Excel helper, not a bulk-table editor, and not a one-off rule tool.

Its position is:

- A campaign-centric optimization workspace
- A rule-driven decision system
- A draft-first execution system
- A local-first SaaS product that can later evolve into multi-user cloud collaboration

The product transforms Amazon Bulk Operations workflows from low-level spreadsheet manipulation into high-level optimization operations.

### 2.2 Target Users

Primary users:

- Amazon store operators
- PPC specialists
- Advertising managers
- In-house growth teams

Extended users:

- Agency account managers
- Multi-brand advertising analysts
- Performance operations leads
- Future AI-assisted optimization users

### 2.3 Product Goals

The product goals of version 3.0 are:

1. Convert raw bulk workbooks into campaign-oriented workspaces.
2. Build a unified business object model that all modules can share.
3. Separate optimization decision-making from final workbook export.
4. Establish a reusable rule architecture that supports Rule Center, simulation, dashboard filtering, and future AI engines.
5. Replace tree-and-table interaction with dashboard-and-workspace interaction.
6. Introduce true SaaS concepts such as repository, workspace, engine, pipeline, draft, execution, simulation, and history.

### 2.4 Design Philosophy

The product should feel closer to:

- Cursor
- Notion
- Pacvue

It should feel less like:

- A spreadsheet parser
- A nested table management panel
- A legacy admin backend

The product experience must prioritize:

- object clarity
- state continuity
- operation safety
- optimization confidence
- reusable system structure

### 2.5 Core Principles

The system follows these principles:

1. `Workbook` is input, not product structure.
2. `Campaign` is the primary operational object.
3. `Workspace` is the primary interaction container.
4. `Rule` produces `Draft`, not direct mutation.
5. `Draft` is the only editable execution proposal.
6. `Export` writes validated patches back to a workbook copy.
7. All engines must share a common context model.
8. All user-facing modules must use the same object language.

## 3. Business Architecture

### 3.1 Business Objects

The product uses a unified business object system.

Core objects:

| Object | Role | Description |
| --- | --- | --- |
| Workbook | Input | Uploaded Amazon Bulk file, preserved as source of truth for export patching |
| Campaign | Primary business object | The top-level optimization object shown in dashboard and workspace |
| Workspace | Interaction container | Independent operating context for a campaign |
| Keyword | Optimization unit | Search keyword entity from bulk data |
| Product Target | Optimization unit | Product targeting entity from bulk data |
| Recent Advertising Data | Decision reference | Recent performance dataset used for rule context only |
| Rule | Decision logic | Structured optimization logic with conditions and actions |
| Draft | Execution proposal | Reviewable adjustment plan generated by rule engine or AI |
| Execution | Finalized action | Approved draft applied during export or later cloud execution |
| Lifecycle | Strategy grouping | A classification used to load rule sets and monitor campaign state |
| Repository | SaaS container | Future persistent storage for rules, workspaces, histories, templates |

### 3.2 Campaign Hierarchy

The new product hierarchy is:

```text
Workbook
  -> Campaign Dashboard
    -> Campaign Workspace
      -> Keyword / Product Target
        -> Rule Evaluation
          -> Draft
            -> Execution
```

This hierarchy deliberately replaces the old structure:

```text
Workbook
  -> Sheet
    -> Table
      -> Row
```

The old structure exposed storage format to users.

The new structure exposes business intent to users.

### 3.3 Workspace Concept

Each campaign owns a dedicated workspace.

A workspace is not just a page. It is a bounded operational context containing:

- campaign identity
- current bulk-derived entities
- recent advertising data scope
- lifecycle assignment
- rule applicability
- draft list
- history
- settings
- local state

Each workspace can be opened in tabs and resumed later.

### 3.4 Lifecycle Management

Lifecycle is a strategic grouping layer, not a physical data layer.

Its purposes are:

- load rule bundles
- group campaigns by optimization strategy
- power dashboard segmentation
- support bulk operations
- support future AI policy selection

Lifecycle does not directly modify campaign data.

### 3.5 Optimization Workflow

The target business workflow is:

```text
Upload Workbook
  -> Parse Workbook
  -> Build Campaign Dashboard
  -> Open Campaign Workspace
  -> Match Recent Advertising Data
  -> Assign Lifecycle
  -> Load Rule Context
  -> Execute Rule Engine / AI Engine
  -> Generate Draft
  -> Review and Approve
  -> Export Workbook Patch
  -> Save Execution History
```

This workflow makes the product feel like software, not file handling.

## 4. System Architecture

### 4.1 Overall Architecture

The system is composed of five layers:

1. Presentation Layer
2. Workspace Layer
3. Processing Layer
4. Optimization Layer
5. Export and Persistence Layer

Architecture overview:

```text
Presentation Layer
  Dashboard / Workspace / Rule Center / Draft Review / Settings

Workspace Layer
  Campaign Workspace State / Tabs / Filters / Context / History

Processing Layer
  Workbook Parser / Recent Data Parser / Normalizer / Aggregator

Optimization Layer
  Rule Repository / Rule Context / Rule Engine / AI Engine / Simulation

Export & Persistence Layer
  Draft Validator / Workbook Patch Engine / IndexedDB Repository / Execution Log
```

### 4.2 Module Architecture

Major modules:

- Bulk Import
- Campaign Dashboard
- Campaign Workspace
- Lifecycle Management
- Rule Center
- Rule Engine
- Simulation
- Draft Center
- Export Engine
- Dashboard Analytics
- Repository Layer
- Settings

Each module must use the same business objects and naming conventions.

### 4.3 Engine Pipeline

The optimization pipeline is redefined as:

```text
Rule Repository
  -> Rule Context
    -> Rule Engine
      -> Draft
```

This replaces the weaker version where rules were treated as page-level behavior.

Benefits of the new pipeline:

- Rule Center can manage rule assets centrally.
- Dashboard filters can reuse condition logic.
- Simulation can reuse engine inputs and outputs.
- AI can plug into the same context and draft contract.
- Codex can generate implementation with much less ambiguity.

### 4.4 Data Flow

Primary data flows:

1. Workbook ingestion flow
2. Recent data ingestion flow
3. Rule evaluation flow
4. Draft review flow
5. Export flow
6. Local persistence flow

Data flow rule:

- source data is immutable
- computed context is derived
- draft data is reviewable
- export output is patch-based

### 4.5 Storage Design

Version 3.0 uses local-first persistence and future SaaS-compatible repository design.

Storage layers:

- in-memory workspace state
- IndexedDB local repository
- workbook binary cache
- recent data cache
- rule repository cache
- execution log cache

Future cloud mapping:

- local repository -> tenant repository
- local workspace -> shared workspace
- local history -> execution audit log
- local rules -> versioned rule registry

## 5. Unified Data Model

### 5.1 Model Principles

All modules must use the same object language.

Forbidden modeling direction:

- page-specific objects
- table-only terminology
- sheet-driven UI naming
- parser-driven business semantics

Required modeling direction:

- campaign-centric
- workspace-centric
- engine-compatible
- execution-safe
- AI-extensible

### 5.2 Core Entity Set

The core entities of the product are:

- Campaign
- Workspace
- Keyword
- Product Target
- Recent Advertising Data Row
- Lifecycle
- Rule
- Rule Context
- Draft
- Execution Log

### 5.3 Workbook as Input

`Workbook` remains important but only as:

- import source
- row locator source
- export patch target
- audit reference

It must not define the product's visible interaction hierarchy.

### 5.4 Campaign

Campaign is the top-level operating object presented to users in dashboard and workspace views.

A campaign should contain:

- campaign identity
- lifecycle assignment
- KPI snapshot
- optimization counts
- recent update state
- workspace pointer

Functional requirement:

- A campaign can be opened as a workspace.
- A campaign can participate in dashboard analytics.
- A campaign can be assigned to a lifecycle.
- A campaign can load its rule context independently.

### 5.5 Workspace

Workspace is the main operating unit.

A workspace contains:

- current campaign summary
- tabs
- overview state
- keyword state
- product target state
- recent data state
- rules state
- draft state
- history state
- settings state

Functional requirement:

- Each workspace preserves its own UI and data state.
- Tabs can reopen previous workspaces.
- Workspace state survives refresh through local persistence.

### 5.6 Keyword

Keyword is a rule-evaluable optimization object.

Minimum business fields:

- keyword text
- match type
- current bid
- status
- bulk metrics
- recent metrics
- source row locator

### 5.7 Product Target

Product Target follows the same optimization contract as Keyword wherever possible.

Design goal:

- reuse engine interfaces
- avoid separate product logic unless required

### 5.8 Recent Advertising Data

Recent advertising data is a decision-support dataset.

It is never treated as export source.

Its roles are:

- provide short-term performance context
- support recent-vs-bulk comparison
- support share metrics and rank logic
- power rule conditions and simulation

### 5.9 Rule

Rule is a reusable optimization asset.

A rule contains:

- metadata
- scope
- lifecycle binding or shared binding
- condition group
- action group
- priority
- safety constraints
- version data

### 5.10 Draft

Draft is the only user-reviewable proposal object.

A draft may be generated by:

- rule engine
- AI engine
- manual action
- simulation result adoption

Draft requirements:

- traceable source
- row locator integrity
- old value and new value
- reason
- matched rule or generator
- selectable status
- validation state

### 5.11 Execution Log

Execution log records:

- when a draft was generated
- by which engine
- with which rule set
- under which workspace context
- what was selected
- what was exported
- whether export succeeded

## 7. Bulk Import System

### 7.1 Design Goal

The import system must convert a raw workbook into structured campaign workspaces without forcing users to think in workbook internals.

### 7.2 Functional Requirements

The system must:

1. Accept Amazon Bulk Operations workbook upload.
2. Preserve the original workbook binary for future export patching.
3. Parse target sheets asynchronously.
4. Identify campaign, ad group, keyword, product target, bid, state, and row locator fields.
5. Build normalized campaign objects and workspace-ready datasets.

### 7.3 Data Source

Primary source:

- Amazon Bulk Operations workbook

Optional auxiliary source:

- recent advertising CSV

### 7.4 Workflow

```text
Upload Workbook
  -> Validate File Type
  -> Read Workbook Buffer
  -> Parse Target Sheets in Worker
  -> Normalize Rows
  -> Build Campaign Index
  -> Build Workspace Data
  -> Aggregate KPIs
  -> Persist Snapshot
```

### 7.5 Interaction Requirements

The UI must show:

- upload area
- parsing progress
- file status
- row count
- sheet summary
- warnings
- success state

### 7.6 State Management

Import state includes:

- idle
- uploading
- parsing
- parsed
- failed

### 7.7 Error Handling

The system must handle:

- unsupported file format
- corrupted workbook
- missing required columns
- empty target sheets
- excessive memory pressure

### 7.8 Performance Requirements

The parser must support large files and keep the UI responsive.

Target direction:

- 20,000 to 100,000+ rows
- worker-based parsing
- incremental ingestion
- progressive KPI aggregation

### 7.9 Future Extension

Future support may include:

- multi-sheet ad-type parsing
- cloud import jobs
- template-based field mapping
- automatic workbook version detection

## 11. Rule System

### 11.1 Design Goal

The rule system is no longer just an engine implementation detail.

It becomes a product-level platform capability that powers:

- Rule Center
- Rule Engine
- Simulation
- Dashboard filter reuse
- AI compatibility

### 11.2 Target Architecture

The official rule pipeline is:

```text
Rule Center
  -> Rule Repository
  -> Rule Context
  -> Rule Engine
  -> Draft
```

### 11.3 Rule Repository

Rule Repository stores:

- rule definitions
- metadata
- version history
- scope bindings
- lifecycle bindings
- enablement state

### 11.4 Rule Context

Rule Context unifies everything the engine needs to evaluate a campaign entity.

It may include:

- bulk metrics
- recent metrics
- comparison metrics
- derived metrics
- lifecycle info
- workspace scope
- safety constraints

### 11.5 Rule Engine

Rule Engine responsibilities:

- load applicable rules
- evaluate condition groups
- generate actions
- convert actions to drafts
- prevent unsafe direct mutation

### 11.6 Conflict Resolution

If multiple rules hit the same optimization unit:

- priority ordering applies
- conflict rules must be deterministic
- the draft layer must preserve reason traceability

### 11.7 Safety Rules

The system should support safety rules such as:

- minimum bid floor
- maximum bid ceiling
- no change when data is insufficient
- no export without valid row locator
- no action when match confidence is ambiguous

### 11.8 Functional Requirements

The rule system must support:

- nested condition groups
- multiple metrics
- multiple data sources
- prioritization
- lifecycle-based loading
- draft generation
- simulation reuse

### 11.9 Future Extension

Future rule capabilities may include:

- shared rule library
- brand-level templates
- tenant-level governance
- AI-authored rules
