请帮我开发一个“亚马逊物流处理系统”桌面网页应用，目标用户是中文运营人员，要求界面简单直观，支持上传 Excel 和 PDF，自动处理亚马逊发货相关数据，并导出结果文件。

一、项目目标
这是一个面向内部使用的物流处理工具，围绕 4 类 Excel 文件和 1 类 PDF 文件工作：

1. 装箱表 A
2. 发货模板 B
3. 包装箱信息表 C
4. 物流发票模板 D
5. 亚马逊导出的箱唛 PDF

系统需要实现：
1. 上传并解析 A/B/C/D/PDF
2. 自动抽取字段
3. 自动填写 B、C、D
4. 解析 PDF 每页信息
5. 支持按规则重命名 PDF
6. 展示每页 PDF 关键数据
7. 导出处理后的 Excel、PDF、汇总表、物流模板

二、技术要求
1. 前端使用现代 Web 技术开发，推荐 React + TypeScript。
2. 后端可用 Node.js，优先本地文件处理方案。
3. Excel 处理必须稳定，保留原始模板格式、公式、下拉框、样式。
4. PDF 需要支持逐页解析和逐页展示。
5. 所有处理逻辑尽量前后端分层，代码结构清晰。
6. UI 为中文。
7. 必须考虑后续扩展：以后可能新增“赛狐发货表”适配逻辑。
8. 上传后的数据要保存在当前会话状态中，支持用户重复调整和重新导出。
9. 所有导出文件名要有清晰规则。
10. 页面风格要求专业、简洁、偏业务系统风格，不要花哨。

- 如果页面中能提取到 Qty，则提取数量

3. PDF 的“公共信息”来源

货件3-LGB8-FBA19H4WYDTY-7箱-美森正班卡派

- 渠道名

- 总箱数

- FBA箱子编码

5. D 表逐箱生成逻辑
系统需要支持：选择 D 模板后，一键为当前 PDF 生成对应物流模板。

生成方式：
按 PDF 页逐箱生成 D 表中的明细行。

每个箱子的明细字段规则：
- 箱序号：按顺序 1,2,3,4...
- FBAID*：填入该页的 FBA 箱子编码，例如 FBA19H4WYDTYU000001
- POAnumber*：不填
- 单件实重*：填入该箱对应重量，优先用 A/C 表该箱的重量 kg
- 长：默认 50
- 宽：默认 40
- 高：默认 40
- 海关编码*：来自 A 表
- 物品中文名*：来自 A 表
- 物品英文名*：来自 A 表，如果 A 表没有英文名，则需要预留手动编辑或后续维护；当前版本可先优先使用“系统字段/映射表/手工补录”
- 产品申报数量/每箱*：来自该箱对应的装箱数量
- 申报单价*：采购成本 / 8
- 品牌：填“无”
- 型号：填“无”
- 材质(英文)*：默认“尼龙/Nylon”
- 用途(英文)：默认“户外/Outdoor”
- 产品图片：拉取 A 表图片

6. 单件实重字段说明
这里要求填“对应的重量”，当前样本中 D 表现有数据填写的是每箱重量值。
所以当前版本按“箱重量”填入单件实重列，后续可再调整字段含义。

7. 多 SKU 箱处理逻辑
PDF 中有 Mixed SKUs 页面。
对于 Mixed SKUs：
- 需要根据 A/C 表中该箱号下有哪些 SKU 数量 > 0 来确定该箱包含哪些 SKU
- D 模板一箱是否写一行还是多行，要按业务要求设计为“一个箱子可展开成多条明细行”
- 即：同一箱如果含多个 SKU，则 D 表中该箱序号可重复多行，每个 SKU 一行
- 同一箱的 FBAID、长宽高、重量相同
- 每条明细的海关编码、中文名、英文名、申报数量、申报单价按该 SKU 分别填写

8. Single SKU 箱处理逻辑
若某页是 Single SKU，且能解析出 SKU 或 Qty，则优先用 PDF 页面数据校验 A/C 结果。
最终以 A/C 装箱数据为主，PDF 为辅助校验。

9. 英文名策略
A 表当前样本没有现成英文名字段，所以需要设计：
- 第一优先：如果系统已有 SKU 对应英文名缓存，则使用缓存
- 第二优先：尝试从既有模板 D 样本中同 SKU 历史记录反查
- 第三优先：允许用户在页面手动补录后生成
- 如果没有英文名，生成前给出提示，允许继续但标记缺失

10. 图片策略
A 表“产品图片-不嵌入”列可能是文字、链接或空值。
系统需要兼容：
- 图片链接
- 本地无图片
- 纯文本描述
若无法真正嵌入图片，则在 D 表中保留文本链接或空白，并给出提示。

四、装箱数据汇总表导出
系统要支持按现有数据生成一个“装箱数据汇总表”，格式参考用户提供的补货表风格。

需要输出内容至少包括：
- 重量/kg
- 数量
- 仓库名称
- 货件号
- SKU
- 最终发货
- 各箱号 1,2,3,4...n
- 物流渠道
- 物流单价
- 物流费用
- 入库配置费（美金）
- 总费用

规则说明：
1. 重量/kg：来自箱号对应重量
2. 数量：来自每箱每 SKU 数量
3. 仓库名称：来自 PDF 或模板
4. 货件号：来自 PDF 文件名或解析结果
5. SKU：来自 A/C
6. 最终发货：来自 A 表发货总数
7. 各箱号列：来自 A/C 装箱明细
8. 物流渠道：默认允许手工填写或从 PDF 文件名解析初值
9. 物流单价：允许手工填写
10. 物流费用：公式计算，建议 = 重量 * 物流单价
11. 入库配置费：允许手工填写
12. 总费用：公式汇总

注意：
这张汇总表需要可下载，格式整齐，适合业务人员直接使用。

六、核心处理规则
1. 所有匹配以 SKU 为主键。
2. 箱号映射必须按真实列头匹配，不要假设从第1箱连续。
3. A 表最后一个 sheet 为准。
4. 发货总数为 0 的 SKU 不写入 B 表。
5. 生成 C/D 时要按箱号维度处理。
6. Mixed SKUs 箱必须支持拆多行。
7. 所有生成结果保留原模板格式。
8. 缺失字段不能直接报错退出，要给出“缺失提示 + 可继续处理”的机制。
9. 用户需要能预览处理结果后再下载。
